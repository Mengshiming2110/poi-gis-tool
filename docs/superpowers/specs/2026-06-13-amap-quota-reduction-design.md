# 高德 API 限额优化设计

## 背景

当前移动端 POI 采集使用高德 Web 服务 `place/around`。免费额度有限，固定网格乘以类别再翻页会快速消耗每日调用次数。触发限额后继续请求没有意义，还会让用户误以为网络或程序故障。

## 目标

- 今天配额用完时，立即停止并给出明确提示。
- 重复区域不重复调用高德，优先使用本地缓存。
- 对高密度网格减少盲目翻页，改为小范围探测。
- 在采集界面展示大致请求预算，让用户知道调用成本。

## 当前实现

- 每个 `类别 + 中心点 + 半径` 的结果缓存到 `localStorage`，缓存有效期 30 天。
- 采集开始前检查当天该 REST Key 是否已触发限额；已触发则不再发请求。
- 高德返回日配额用完、Key 无效、Key 类型不匹配时立即停止后续请求。
- 默认每个查询最多请求 2 页，可通过 `localStorage.amap_max_pages_per_query` 调整。
- 当第一页返回数量超过阈值时，优先把当前网格拆成 4 个子格各查第一页，而不是继续翻大量页。
- 采集页显示估算调用次数，并提示缓存命中会自动跳过。

## 后续方向

- 把缓存从手机本地升级到 Supabase 区域缓存，实现手机和桌面共享缓存。
- 在网格步骤增加“预计调用次数”和“推荐网格大小”。
- 对公开数据源如 OSM 增加补充层，用于低成本公共设施数据。
